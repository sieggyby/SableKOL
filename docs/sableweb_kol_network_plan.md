# SableWeb KOL network viewer — integration plan

**Date:** 2026-05-07
**Status:** Plan; no code yet.
**Companion:** `sablekol_generalization_plan.md` — backend generalization
  must land before this UI is meaningful for non-SolStitch clients.
**Target surface:** `/ops/kol-network` (operator-only).
**Stack:** Next.js 16.2.1 App Router · d3-force (already in package.json) ·
  Postgres via existing `DbDriver` · Tailwind · operator-role auth via
  existing `getSession()` middleware.

---

## What we're moving from local HTML to a hosted page

The standalone `~/Downloads/solstitch_network_2026_05_07_*.html` artifact
is operator-laptop-only, vis.js-rendered, single-tenant. The SableWeb
version becomes:

* multi-client (operator picks client from a dropdown);
* live (reads `kol_extract_runs` / `kol_follow_edges` / `kol_candidates`
  via the same `DbDriver` used by `/ops`);
* role-gated (operator email allowlist; clients never see this);
* version-controlled (no local file artifacts to email around);
* embeddable (clients with very specific permissioning could see a
  filtered subset, but that's Phase 3 — *not* in initial scope).

## Architecture (matches existing SableWeb idioms)

### 1. Data layer

**Postgres reads (live mode):**
```sql
-- per client_id (the org_id used by SableKOL — '_external' for SolStitch
-- since it's path-(ii) external, otherwise the actual sable.db org_id)
SELECT ... FROM kol_extract_runs WHERE ... AND cursor_completed = 1;
SELECT ... FROM kol_follow_edges WHERE run_id IN (...);
SELECT ... FROM kol_candidates WHERE handle_normalized IN (...);
```

**Demo mode:** static fixture from `src/data/kol_network_solstitch.json` —
generated by a one-shot `scripts/build_kol_network_fixture.py` from
SableKOL's local sable.db. Updated when the operator wants the demo to
reflect new state.

### 2. Server route

`/api/kol-network/[clientId]/route.ts`:

```ts
GET /api/kol-network/solstitch?role_filter=...&size_bucket=...&color=...

Returns: {
  client: { id, display_name, mode, debut_date, themes },
  cohort: [{ handle, twitter_id, completed_at }],
  candidates: [{ handle, tier, score, vibe_fit, ... }],
  kingmakers: [{ handle, in_pool, is_org, is_celebrity, ... }],
  edges: [{ from, to }],          // restricted to query result
  stats: { embedded_nodes, embedded_edges, last_updated_at }
}
```

Implements:
- The same `is_organization` / `is_celebrity` / `vibe_fit` filters from the
  Python code (mirrored in TS for SSR cheapness — denylists are config
  files at `~/.sable/clients/<id>.yaml`, exposed via `loadClientConfig()`).
- Server-side pre-trim by score so payload stays under ~1MB (the slider
  in the client never needs more than ~5K nodes).
- 5-minute cache via Next.js `revalidate: 300`.

### 3. Client component

`src/components/ops/KOLNetwork.tsx` — `'use client'`, mirrors
`RelationshipGraph.tsx` patterns:

* d3-force simulation (already a SableWeb dep, no new package)
* Canvas rendering for >1K nodes (SVG hits ~1.5K-node performance ceiling)
* Tailwind-styled control panel (matches `/ops` design tokens —
  `#0A0A0C` background, `#C8A86E` accent, mono for stats)
* Handles all the controls from the standalone HTML:
  - Network-size slider (Least → Most)
  - Role / tier / sector filters
  - Spread / size / pulse sliders
  - Org + celebrity toggles (default off)
  - Search
  - Click-highlight, shift+click → x.com profile
  - Color modes (in_pool / sector / tier / role / vibe_fit)
* Adds web-only features:
  - **Save view** — persist current filter state per operator
  - **Permalink** — current state encoded into URL, so an operator can
    paste the link to a colleague
  - **Compare clients** — side-by-side two networks (Phase 2)
  - **Annotation layer** — operator notes attached to specific nodes,
    visible to other operators (Phase 2)

### 4. Page

`src/app/ops/kol-network/page.tsx` — server component:
- Auth gate via `getSession()` + `isOpsRole(session.role)` (matches `/ops`)
- Server-fetches the default client's data
- Renders `<KOLNetwork>` with initial props
- Adds a `<ClientPicker>` (existing `OpsConsole`-style dropdown) for
  switching clients

### 5. Navigation

Add to `OpsDotNav` and the morning-brief / nav structure under a new "KOL"
section.

## Plan (phases)

### Phase 1 — fixture-mode demo (~1-2 days)

Goal: a working demo on `sable-web-iota.vercel.app` showing the SolStitch
network. No live DB read yet.

1. Write `scripts/build_kol_network_fixture.py` in SableKOL — exports the
   same node + edge JSON the local HTML embeds, but written to
   `~/Projects/SableWeb/src/data/kol_network_solstitch.json`.
2. Build `KOLNetwork.tsx` React component using d3-force. Port the JS
   logic from the local HTML one-for-one.
3. Build the page, route, server fixture loader.
4. Smoke-test on Vercel demo deployment.

### Phase 2 — live-mode (Hetzner Postgres) (~2-3 days)

Goal: data pulled live from sable.db on every load (with 5-min ISR cache).

1. Wait for SableKOL Phase-2 generalization (per-client config) — required
   for the API route to know which audiences/cohorts/sectors to serve per
   client.
2. Add `loadKOLNetwork(clientId)` to `src/lib/data-loader.ts` — same
   pattern as existing `loadAllClientsAsync()`.
3. Mirror Python filters (`is_organization`, `is_celebrity`, vibe-fit)
   into `src/lib/kol-filters.ts`. Tests against fixture parity.
4. Wire the API route. Cap payload at ~1MB / 5K nodes.
5. Switch the page from fixture to API call (still SSR-rendered).

### Phase 3 — operator features (~1-2 days)

1. **Save view per operator** — `/ops/kol-network/views` listing pinned
   filter states. Stored in `op_pinned_views` table (new — Sable Platform
   migration).
2. **Permalink-as-state** — encode current filter state in URL search
   params (already half-done by the size_bucket / role_filter query
   params).
3. **Annotation layer** — operator text notes attached to handles,
   visible to other operators with role ≥ operator. Stored in a new
   `kol_operator_notes` table.
4. **Compare two clients** — side-by-side Cosgrove view, identical
   controls, separate datasets. Useful for cross-client kingmaker spotting.

### Phase 4 — client-facing window (later, NOT initial scope)

Some clients may eventually be allowed to see a heavily-redacted version
of their own network — e.g., a hidden-gem candidate list with the operator
score column hidden. **Per the SableWeb data-exposure rules** (CLAUDE.md
§ "Clients NEVER see"), this requires:
- Stripping `fit_score`, `score`, `cost_usd`, etc.
- Hiding cohort/audience-source provenance
- Renaming Tier-A/B/C labels (cluster numeric tiers map to client-friendly
  language)

Defer until at least one client explicitly asks for it. Easier to keep
the whole tool ops-only.

## Risks / non-trivial decisions

* **Bundle size.** d3-force is fine; if we end up needing vis-network for
  any reason (e.g., its physics is better-tuned than d3's for our shape),
  it's ~150KB gzipped — acceptable but not free. Default to d3-force.
* **Canvas vs SVG.** SVG has nicer interaction but stalls past ~1500
  nodes. We cap at 5K → canvas mandatory. Hover/click interaction needs a
  manual hit-test (loop over nodes, distance check) — straightforward.
* **Live mode dependence on Phase-2 generalization.** Without per-client
  config files, the API route would have to know SolStitch handles by
  name. Not blocking Phase 1 fixture demo.
* **Postgres query shape.** A naïve query for "all edges between visible
  nodes" returns up to 134K rows for the SolStitch dataset. Solution: the
  API does the score-trim server-side (≤5K nodes) and queries edges only
  between those. Query becomes `WHERE follower_handle IN (?) AND
  followed_handle IN (?)` over the trimmed set.
* **Data freshness.** ISR 5-minute cache is enough — KOL data changes
  weekly at most. Manual cache-bust via `revalidate` action button on the
  page header.
* **Privacy / secondary clients.** TIG / Multisynq / PSY may not want
  their cohorts visible to each other's operators. Default ACL: every
  operator sees all clients. If needed later, add `client_acl` per-user
  list to `~/.sable/allowlist.json`.
* **Rendering performance for the pulse animation.** d3-force already
  re-tickles every frame; layering pulse on top of canvas redraws is
  cheap. Can disable when tab hidden via `IntersectionObserver`.

## Acceptance criteria

* Phase 1 done when:
  - `/ops/kol-network` renders in demo mode on Vercel
  - Default view shows SolStitch network at "Less" density
  - All filter controls work
  - Click + shift+click + search work
  - Org + celebrity filters default ON

* Phase 2 done when:
  - Live mode reads from Postgres
  - Adding a new client (per Phase-2 generalization) makes its network
    available without code changes
  - Page renders in <2s server-side, <500ms client-side after warm cache

* Phase 3 done when:
  - Saved views persist across operator sessions
  - Permalinks reproducibly restore filter state
  - Annotation layer visible to other operators

## Operator workflow (post-launch sketch)

1. Operator visits `/ops/kol-network`
2. Picks client from dropdown (default = most recently active client)
3. Adjusts size, filters; sees the people-only network
4. Clicks a node — highlights neighbors, sees stats card on the right
5. Shift+clicks → opens x.com profile in new tab to scout the bio
6. Adds an annotation ("knows Alice personally; warm intro available")
7. Saves the current filter state as a named view ("Stealth top-50 fashion")
8. Shares the permalink with another operator

This replaces the operator's current workflow of regenerating local
HTML files via Python scripts and emailing them around.
